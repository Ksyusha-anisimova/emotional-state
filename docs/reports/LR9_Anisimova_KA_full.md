1. Теоретическая часть
1.1. CI/CD
CI/CD - это практика автоматизации этапов проверки и доставки программного кода. В рамках CI при каждом изменении запускаются проверки, а в рамках CD код подготавливается к выпуску.

1.2. Роль тестирования в конвейере
Тестирование в конвейере CI/CD позволяет рано обнаруживать дефекты и не допускать их попадания в основную ветку. Для этого в пайплайн включают линтинг, статический анализ, модульные и интеграционные тесты.

1.3. Статический анализ
Статический анализ проверяет код без его выполнения. Он помогает найти нарушения стиля, потенциальные ошибки и слишком сложные участки кода.

1.4. Покрытие тестами
Покрытие кода тестами показывает, какая часть кода действительно исполняется во время тестов. Эта метрика не гарантирует отсутствие дефектов, но помогает видеть непроверенные участки и контролировать качество регрессии.

4. Контрольные вопросы
1. Что такое CI/CD и каковы его основные компоненты?
CI/CD - это подход к автоматизации сборки, проверки и доставки кода. Основные компоненты: pipeline, stage, job, runner, artifacts, cache.

2. Чем отличаются Continuous Delivery и Continuous Deployment?
В Continuous Delivery код готов к выпуску после автоматических проверок, но решение о релизе принимается вручную. В Continuous Deployment релиз выполняется автоматически после прохождения всех проверок.

3. Что такое pipeline, stage и job?
Pipeline - весь конвейер проверок. Stage - логический этап пайплайна. Job - конкретная задача внутри этапа.

4. Какие преимущества дает тестирование в CI/CD?
Быстрое обнаружение дефектов, единые правила качества, снижение риска регрессии и более стабильные релизы.

5. Чем artifacts отличаются от cache?
Artifacts - результаты конкретного запуска (отчеты, логи), которые нужно сохранить. Cache - техническое кэширование зависимостей для ускорения следующих запусков.

6. Что такое статический анализ кода?
Это проверка исходного кода без выполнения программы с целью поиска дефектов, нарушений стиля и проблем поддержки.

7. Чем статический анализ отличается от динамического тестирования?
Статический анализ не исполняет код. Динамическое тестирование выполняет программу и проверяет фактическое поведение.

8. Что такое code smell?
Это признак неудачной структуры кода, который усложняет поддержку. Например, длинный метод, дублирование логики, высокая связность.

9. Что такое цикломатическая сложность?
Это метрика количества независимых путей выполнения. Чем она выше, тем труднее тестировать и сопровождать код.

10. Какие инструменты статанализа использованы в работе?
ESLint, Lizard и CodeQL.

11. Что такое coverage и какие виды покрытия бывают?
Coverage - доля кода, выполняемая тестами. Обычно измеряют coverage по строкам, ветвлениям, функциям и инструкциям.

12. Какие инструменты для coverage использовались?
В проекте использован c8.

13. Гарантирует ли 100% coverage отсутствие ошибок?
Нет. Coverage показывает факт выполнения кода, но не гарантирует корректность всех проверок.

14. Для чего используются Allure Reports?
Для визуализации и хранения истории результатов тестовых запусков. В данной работе Allure не подключался.

15. Как публиковать отчеты через GitHub Pages?
Нужно сгенерировать статический отчет в workflow и настроить публикацию папки отчета в GitHub Pages.

16. Как настроить branch protection с обязательными CI-проверками?
В Settings -> Branches создается правило для `main`, после чего включаются required status checks.

17. Что такое CodeQL и зачем он нужен?
CodeQL - инструмент статического анализа безопасности от GitHub, который ищет потенциальные уязвимости и небезопасные конструкции.

18. Как хранить токены и пароли в CI/CD?
Только в защищенных секретах репозитория (Secrets/Variables), без хранения в открытом коде.

on:
  push:
    branches:
      - "**"
  pull_request:
    branches:
      - main
      - master
jobs:
  test:
    name: test (lint + static + unit)
    runs-on: ubuntu-latest
  integration:
    name: integration (BDD)
    runs-on: ubuntu-latest
  report:
    name: report (artifacts + quality)
    runs-on: ubuntu-latest
